# d12 Block Attention Residuals (AttnRes)

**Date:** 2026-03-19
**Branch:** `exp/attention_improvements`
**Hardware:** Single H100 80GB HBM3
**Wall time:** 60.01 min
**Paper:** arXiv:2603.15031 (MoonshotAI/Kimi)

## Summary

Implemented Block Attention Residuals, replacing the fixed residual accumulation
(`resid_lambdas` + `x0_lambdas` + `backout_lambda`) with softmax attention over
depth. Each sub-layer (attention, MLP) gets a learned pseudo-query to selectively
aggregate previous block representations.

**Result: Clear failure.** Final BPB **1.016** vs baseline **0.854** (+0.162, 19% worse).
Throughput dropped 42% (546k → 320k tok/sec), wall time nearly doubled (35 → 60 min),
and memory usage increased 55% (29 → 45 GB). Quality never caught up despite the
additional compute.

## Implementation Details

- Replaced `resid_lambdas`, `x0_lambdas`, and `backout_lambda` with `attn_res_queries`
- `attn_res_num_blocks`: auto = `n_layer // 3` = 4 blocks for d12 (3 layers per block)
- 2 queries per transformer layer (pre-attn + pre-MLP) + 1 final output query = 25 total
- Zero-init pseudo-queries (starts as uniform averaging = standard residual)
- RMSNorm on keys to prevent magnitude domination
- `_attn_res()`: `torch.stack(sources)` → `rms_norm` → `einsum` → `softmax`

## Why It Failed

### 1. Throughput death by a thousand cuts
`_attn_res()` is called 25 times per forward pass (2× per layer × 12 layers + 1 final).
Each call does `torch.stack(sources)` + `rms_norm` + `einsum` + `softmax`. This is
pure memory traffic — rebuilding Python lists and restacking tensors every sublayer.
The overhead is ~700ms per step (960ms baseline → 1640ms AttnRes).

### 2. Sub-layer granularity is too aggressive
The paper uses Block AttnRes with N≈8 blocks and one query per block boundary.
This implementation uses 2 queries per transformer layer (pre-attn + pre-MLP),
which is 25 queries for d12. The paper's approach would be ~8 queries — 3x fewer
depth attention calls.

### 3. Removed proven mechanisms without replacement
The implementation removed `resid_lambdas`, `x0_lambdas`, and `backout_lambda` —
all of which were contributing to the baseline's 0.854 BPB. The AttnRes mechanism
needs to learn from scratch what these scalars already provided, starting from
uniform averaging (zero-init).

### 4. d12 is too shallow for the overhead to pay off
The paper's scaling results show AttnRes gains grow with depth. At d12 (12 layers,
4 blocks), the depth signal dilution problem is mild. The overhead of softmax
attention over depth costs more than the signal it preserves.

## Throughput Diagnosis (from Codex analysis)

- FA3 is confirmed active — NOT an SDPA fallback issue
- `torch.compile` captures the full forward in one graph, zero graph breaks — NOT a Dynamo issue
- The bottleneck is kernel count and memory movement in the AttnRes loop
- Potential fixes (not tested):
  1. Cache completed blocks as stacked tensor instead of rebuilding lists
  2. Batch inter-block queries (precompute once per block for all 6 queries)
  3. Reduce `attn_res_num_blocks` from 4 to 2
  4. Paper-faithful two-phase block schedule

## Recommendations

1. **Do not use this implementation at d12.** The quality and throughput losses
   are too severe.
2. **If revisiting AttnRes:** restore `resid_lambdas`/`x0_lambdas` as the base
   residual mechanism and add AttnRes as a lightweight supplement (block-level
   only, ~8 queries, not sub-layer).
3. **Consider DCA instead** — it modifies Q/K/V composition rather than residual
   connections, which is complementary to the existing scalar approach and may
   have lower overhead.
4. **AttnRes might be worth testing at d24+** where depth signal dilution is
   more severe, but only with the throughput optimizations above.
