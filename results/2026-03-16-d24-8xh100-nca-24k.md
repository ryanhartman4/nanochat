# Nanochat D-24 Training Run Summary (2026-03-15/16)

## Setup

- **Hardware:** 8x H100 80GB SXM node
- **Disk workaround:** Container has 16GB overlay filesystem, insufficient for training data. Symlinked `~/.cache/nanochat → /dev/shm/nanochat_cache` (1TB RAM-backed tmpfs) for all data shards, checkpoints, tokenizer, and NCA data.
- **Dataset:** ClimbMix-400B, 170 train shards + 1 validation shard (~15GB)
- **Python env:** Installed on overlay via `uv sync --extra gpu`, plus `torchao==0.15.0` for FP8 support

## Configuration

- **Model:** D-24 (1.2B total params, ~730M scaling params), 1536 model_dim, 12 heads
- **Tokenizer:** 24K vocab (vs Karpathy's 32K default)
- **NCA pre-pre-training:** 500 steps, alphabet size 4, LR 3e-4, batch size 8
- **Main training:** FP8, device-batch-size=16, target-param-data-ratio=8.0
- **Total steps:** 5472

## Issues Encountered

1. **`torchrun --run` flag conflict:** `--run=` is ambiguous with torchrun's `--run-path`/`--run_path`. Fix: drop the flag (wandb logging only) or use `python -m torch.distributed.run` instead.
2. **NCA OOM:** Default `--nca-batch-size=32` OOMed at D-24. Fix: `--nca-batch-size=8`. The `--device-batch-size` flag only controls the main training loop, not NCA.
3. **Main training OOM at batch size 32:** D-24 with FP8 uses ~77.7GB per GPU. Batch size 32 tries to allocate 6GB more than available. Batch size 32 does NOT fit at D-24 — stuck at 16 with gradient accumulation of 2.
4. **`TORCH_DISABLE_ADDR2LINE=1`:** Required to suppress noisy C++ stack trace symbolization warnings that flooded the terminal.
5. **Disk full:** `torch.compile` cache (`/tmp/torchinductor_root`) consumed 6.9GB on the overlay. Fix: periodically clear it. Also route tee logs to `/dev/shm` instead of `/tmp`.

## Key Findings

1. **CORE 0.2542 in 95 minutes** — just below the GPT-2 threshold of 0.2565 and the leaderboard best of 0.2626 (99 min).
2. **Batch size 32 doesn't fit at D-24** — both D-24 and D-26 are stuck at batch size 16, so no step speed advantage from choosing the smaller model.
3. **NCA hypothesis is unproven but not disproven** — without a non-NCA baseline at the same config, we can't isolate NCA's contribution. The 0.2542 CORE in 95 min is faster than Karpathy's 99 min but lower CORE.
4. **24K vocab may be limiting CORE ceiling** — smaller vocab means less semantic content per 2048-token context window. Each step is less effective at learning long-range patterns. The D/V gradient flow improvement from 24K may not compensate.
5. **CORE accelerates late** — gains per 500 steps: 0.07 → 0.019 → 0.009 → 0.012 → 0.018 → 0.006. Not a smooth curve; it can re-accelerate once BPB drops below ~0.77.

## Next Steps

- **Retrain tokenizer at 32K vocab** — more bytes per token means more content per sequence, potentially faster CORE convergence
- **Run D-24 + NCA + 32K vocab + FP8 + ratio 8.0** — test whether 32K vocab + NCA together beat Karpathy's 32K-without-NCA by 15+ minutes (target: 84 min or less)
- **If CORE still falls short at ratio 8.0:** try ratio 9.0-9.5 for more training steps, trading some time savings for a higher CORE ceiling
- **Goal:** CORE > 0.2565 in under 84 minutes to demonstrate a meaningful improvement worth adding to the leaderboard
