# Nemotron 3 Super: Synthetic Data for Pre-Training

**Paper:** NVIDIA (2026). "Nemotron 3 Super: Open, Efficient Mixture-of-Experts Hybrid Mamba-Transformer Model for Agentic Reasoning."
**Report:** https://research.nvidia.com/labs/nemotron/files/NVIDIA-Nemotron-3-Super-Technical-Report.pdf
**Data:** https://huggingface.co/collections/nvidia/nemotron-pretraining-specialized-v1.1

---

## Core Idea

Nemotron 3 Super is a 120B total / 12B active parameter MoE hybrid Mamba-Attention model, pre-trained on 25T tokens. The key relevance to our project is their **synthetic data pipeline for pre-training** — they generate multiple categories of synthetic data using large teacher models and mix it into the pre-training corpus, showing measurable benchmark improvements.

## Synthetic Data Categories for Pre-Training

### 1. Synthetic Code Concepts (15M problems)
- Extract 91 programming concepts from HumanEval using GPT-OSS-120B
- Generate problems with GPT-OSS-20B (combine up to 4 concepts per problem)
- Generate 5 solutions per problem with GPT-OSS-120B (≤60 lines)
- Clean via AST validation, filter modified problems
- Result: 15M problem-solution pairs

### 2. Synthetic Unconditional Algorithmic (0.2B tokens)
- Minimalistic prompts ("Write a function") to Qwen3-235B and GPT-OSS-120B
- LeetCode-style problems with random difficulty levels
- GPT-OSS-120B rewrites for edge cases, unit tests, formatting
- Deduplicated via 5-8 word titles generated by GPT-OSS-120B
- Decontaminated against HumanEval, MBPP, CRUXEval, LiveCodeBench
- Small (0.2B tokens) but impactful: +1-2 points on HumanEval, MBPP, CRUXEval-O

### 3. Synthetic Economics
- Multiple-choice questions across micro/macro/econometrics
- Generated by Qwen3-235B-Thinking with step-by-step solutions
- Model-based verification for clarity and accuracy

### 4. Synthetic Formal Logic
- Logic problems: natural language ↔ predicate/propositional logic translation
- Random personas, letters, connectives for variability
- Generated and verified by Qwen3-235B-Thinking

### 5. Synthetic Multiple Choice (3.5M samples, ~1.6B tokens)
- Bootstrapped from MMLU auxiliary training set
- Stage 1: Qwen3-235B generates similar questions from seeds
- Stage 2: DeepSeek-V3 solves each question with reasoning traces
- Majority voting across multiple seeds for answer reliability
- **Ablation result:** Adding 1B tokens of synthetic MCQ to last 100B tokens of 25T training improved MMLU 77.22→77.51, MATH L5 78.55→79.05, AIME-2024 53.3→56.7

### 6. Synthetic Web Crawl (syn-crawl-medium-high, syn-crawl-high)
- Generated from filtered web documents
- syn-crawl-high is **22.4% of Phase 1 data** and 22.4% of Phase 2 — the single largest category
- Quality-tiered: synthetic rewrites of crawl data at different quality levels

## Data Mixture Strategy
- **Phase 1 (80% of 25T = 20T tokens):** Emphasizes diversity. Broad coverage across all categories.
- **Phase 2 (20% of 25T = 5T tokens):** Shifts toward high-quality sources (Wikipedia, finepdfs-high, STEM-SFT). Refines model performance.
- Synthetic data is ~40-50% of the total pre-training mixture (syn-crawl alone is 33.7% in Phase 1)

## Key Technical Details
- **Teachers:** GPT-OSS-120B, GPT-OSS-20B, Qwen3-235B, Qwen3-235B-Thinking, DeepSeek-V3
- **Architecture:** 120B total, 12B active, hybrid Mamba-2 + Attention, LatentMoE (512 experts, top-22), MTP (2 heads)
- **Training:** NVFP4 precision, 25T tokens, WSD schedule, Muon optimizer
- **Context:** 4096 base, extended to 1M

## Key Findings

1. **Synthetic data is a massive fraction of pre-training:** ~40-50% of the corpus is synthetically generated, with syn-crawl-high alone at 22.4%
2. **Small targeted synthetic datasets have outsized impact:** 0.2B tokens of algorithmic code problems gave +1-2 points on coding benchmarks
3. **MCQ synthesis with majority voting works:** 1.6B tokens of MMLU-style synthetic data improved AIME-2024 by 3.4 points
4. **Two-phase curriculum matters:** Diversity first (Phase 1), then quality refinement (Phase 2)
5. **Teacher models used for generation:** The largest available models (120B+) generate, smaller models can't produce reliable training data
6. **Quality filtering is essential:** AST validation for code, model-based verification for reasoning, decontamination against benchmarks

## Relevance to This Project

### Applicability to NEXUS-Sprint: **LOW-MEDIUM**

**The core tension:** Synthetic data generation requires powerful teacher models, and the speedrun measures wall-clock time including all data preparation. Using a 120B model to generate training data for a 1.2B model is not practical within the speedrun constraints.

**Why it's limited for the sprint:**

1. **Wall-clock includes everything.** The speedrun timer captures data prep + tokenizer + training. Generating synthetic data with a teacher model would add significant wall-clock time. nanochat uses pre-existing ClimbMix dataset — no generation step.

2. **Teacher model cost.** NVIDIA uses 120B-235B models for generation. We don't have these available on the H100 node during the speedrun. Pre-generating offline is possible but diverges from the leaderboard's single-script constraint.

3. **Scale mismatch.** The ablation showing +3.4 AIME improvement used 1B tokens of synthetic MCQ added to 100B tokens of continued training on a 25T-token-trained model. nanochat trains on ~10B tokens total. The ratio of synthetic-to-real data would be very different.

4. **ClimbMix is already curated.** nanochat's ClimbMix dataset (from NVIDIA) is already a high-quality curated mix. The marginal benefit of adding synthetic data on top of a curated mix at nanochat's training budget is uncertain.

**Where it could help:**

- **Pre-generate synthetic data offline** (before the speedrun timer starts, if rules allow). A few GB of targeted synthetic code/math/MCQ data mixed into ClimbMix could help CORE score.
- **The 0.2B tokens of algorithmic code** result is intriguing — very small investment, measurable gains. Could we generate a small synthetic dataset once and include it in the data pipeline?
- **For NEXUS-Full:** Synthetic data is a major component of the training recipe. The two-phase curriculum (diversity → quality) is directly applicable.

### For the speedrun specifically:
- **If data is allowed to be pre-generated:** Add ~200M-1B tokens of synthetic MCQ/code to ClimbMix. Potential +1-2 CORE points for minimal training time cost.
- **If data must be generated during the run:** Not feasible — teacher model inference would dominate wall-clock.
- **Check leaderboard rules:** nanochat's rules are flexible on data ("Data: Flexible"). Pre-generated synthetic data should be allowed.

**Estimated integration effort:** 1-2 days to generate synthetic data offline, ~0.5 day to integrate into nanochat's data pipeline. Need to check ClimbMix format compatibility.

**Risk:** LOW if pre-generated, HIGH if attempted during speedrun run.
